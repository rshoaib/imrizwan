# imrizwan.com — SEO Audit, 2026-05-10

This is a delta audit. The 2026-05-08 full audit (296 lines, in `seo-reports/seo-audit-2026-05-08.md`) is the baseline. Two days later, the picture has moved meaningfully: the May 8 critical-path push has shipped, two of the "strategic investment" posts and an entirely new section are drafted but uncommitted, and the same Windows-side friction that almost ate the last batch is starting to recur. Read this in front of the May 8 audit; nothing not mentioned here is unchanged.

## Executive summary

The biggest blocker from the May 8 audit is gone. The recovery PowerShell ran, the eleven staged title/excerpt rewrites landed (`0e31e8b`), the three over-length titles got trimmed (`146f3f3`), four orphan posts got inbound links (`420a12f`), credentials are now `.gitignore`d (`5862697`), and the new Microsoft Graph OData cheat sheet shipped (`f272dcf`). Five clean SEO commits in two days, no lock files, no quarantine directory, working tree healthy on the surface — the pipeline is unblocked.

But almost none of the on-page Critical/High items from the May 8 audit have been addressed. The homepage still says `<h1>Rizwan</h1>`, the four stat counters still SSR as `0+ / 0+ / 0+ / 0`, the `/tools` page still ships zero `ItemList` schema, the `/about` page still has zero `Person` schema and a one-word title, the `viewport` export is still missing from `layout.tsx`, and `dateModified` still mirrors `datePublished` on every blog post. The work that shipped was the *content* recovery; the *template* fixes are all still open.

Meanwhile, three substantial pieces of new work are sitting untracked: the Copilot Studio pillar page (2,824 words), the SPFx vs Power Apps comparison (3,312 words), and an entirely new `/errors` section (Microsoft 365 Error Code Reference with `BreadcrumbList` + `ItemList` JSON-LD baked in). All three were strategic recommendations from the May 8 audit. They're well-built. They just aren't on the site yet.

The new friction is line-ending churn: 18 files show as modified in `git status` but every one of them is a CRLF→LF whitespace-only diff (zero real content change, confirmed via `git diff --ignore-all-space`). If they get committed as-is, they'll bloat history with 4,000 noise lines and obscure the actual content edits underneath. They need to be reverted or normalized before the next push.

Top three priorities this round: (1) commit and push the three drafted assets — they're ready; (2) revert the line-ending-only modifications so the next commit shows real changes; (3) finally execute the on-page template fixes the May 8 audit listed as Critical/High, because no amount of content depth recovers the homepage `<h1>Rizwan</h1>` problem.

Overall assessment: **content side accelerating, on-page template work stalled, mechanical hygiene slipping again**.

## Pre-existing operational issue, recurring: line-ending churn

`git status` shows 18 modified files. Confirmed via `git diff --ignore-all-space --ignore-blank-lines` that the real content delta on every one of them is **zero lines**. Files affected:

- `.gitignore`
- `app/sitemap.ts`
- `app/tools/error-decoder/page.tsx`
- `data/errors.ts`
- `public/rss.xml`
- 13 Microsoft Graph blog posts under `content/blog/`

All are CRLF↔LF normalization changes — likely from a Windows editor saving with CRLF when the file was committed with LF, or vice versa. The total raw-diff line count on these 18 files is roughly 4,000+ lines. If they get committed as-is, the next PR's diff will be unreadable, hide the real changes (the new `copilot-studio-guide.md`, `spfx-vs-power-apps-2026.md`, and `app/errors/`), and fight with `git blame` on every line of every blog post going forward.

Two clean recoveries:

```powershell
# Option A — revert the noise, keep only real changes
cd C:\Projects\imrizwan
git checkout -- .gitignore app/sitemap.ts app/tools/error-decoder/page.tsx data/errors.ts public/rss.xml
git checkout -- (git diff --name-only -- 'content/blog/microsoft-graph-*.md')
```

```powershell
# Option B — set a .gitattributes to normalize once and prevent recurrence
@'
* text=auto eol=lf
*.md text eol=lf
*.ts text eol=lf
*.tsx text eol=lf
*.json text eol=lf
*.xml text eol=lf
'@ | Set-Content .gitattributes -Encoding UTF8
git add --renormalize .
# review with: git diff --cached --stat
git commit -m "chore: normalize line endings to LF (gitattributes)"
```

Option A is the right move *now*, because the actual SEO content you want to push (the two new posts, the new `/errors` section) deserves a clean diff. Option B is the right move *afterwards*, as a one-time normalization commit, so this stops recurring.

This is the second time in a week the same Windows-side hygiene issue has slowed shipping. It's worth solving structurally rather than fighting it per-batch.
